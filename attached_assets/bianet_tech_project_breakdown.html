Two platforms, one ecosystem

Bia'net Tech is not a single app. It's a technology infrastructure layer with two distinct platforms — Seed (B2C/B2B marketplace) and Bartar (commodity exchange). They share auth, user identity, logistics, and AI layers but have separate UIs and business logic.

Total build: ~24 months to full v1
Platforms at a glance

Seed — agricultural marketplace

  • Connects farmers, wholesalers, retailers, exporters, consumers
  • Product listings, verified vendor onboarding, real-time orders
  • 3–5% commission on completed transactions
  • Subscription tiers for advanced seller tools and analytics
  • Sponsored listings and in-app advertising
  • B2C + B2B model, mobile-first

Bartar — commodity exchange

  • Bulk trade coordination — grains, sesame, cassava, cocoa, ginger, rice
  • Cross-border export coordination and NEPC verification
  • 2–4% commission on bulk commodity trades
  • Enterprise dashboards, market intelligence subscriptions
  • Logistics coordination fees for bulk and export shipments
  • B2B focused, enterprise-grade tooling
Shared infrastructure

Identity & verification layer

  • BVN, NIN verification via third-party APIs
  • NEPC export license verification
  • Face recognition for vendor identity
  • Multi-factor authentication (SMS + TOTP)
  • Unified auth used by both Seed and Bartar

Logistics coordination layer

  • Partner logistics provider integration (not owning fleet)
  • Delivery tracking and status webhooks
  • Bulk shipment coordination for Bartar
  • Dispatch coordination API used by both platforms

AI/ML layer

  • Product and trade recommendations (TensorFlow + Python)
  • Pricing analytics and market intelligence
  • Fraud detection and anomaly flagging
  • Demand forecasting for commodity traders

Payments layer

  • Paystack / Flutterwave integration for NGN transactions
  • Escrow logic for large commodity trades
  • Commission auto-deduction on transaction completion
  • Payout schedules for verified vendors

Flutter (mobile) React (web dashboard) Node.js (Express/Fastify) Python (AI/ML + analytics) PostgreSQL MongoDB AWS (primary) Google Cloud (AI services) TensorFlow Python analytics End-to-end encryption MFA Penetration testing
M1

MVP 1 — Seed marketplace (Q1 2027)

The first thing users ever touch. Goal is to onboard verified vendors, enable product listings, process real transactions, and prove the marketplace model works before building complexity. Keep scope ruthlessly small — no AI, no subscriptions, no ads in MVP 1.

Target: Q1 2027 launch → Lagos & Abuja initial
What MVP 1 must do (no more, no less)

Vendor onboarding

  • Vendor registers with phone number + BVN/NIN verification
  • Face recognition liveness check via third-party API
  • Business name, category, location, bank account
  • Admin approval queue — no vendor goes live unreviewed
  • Vendor dashboard: active listings, orders, earnings

Product listings

  • Create, edit, pause, delete product listings
  • Category taxonomy: vegetables, grains, fruits, livestock, equipment
  • Photos (up to 5), description, price, unit, MOQ, location
  • Availability status (in stock / out of stock)
  • No auction in MVP 1 — fixed price only

Buyer experience

  • Browse and search by category, location, price
  • Buyer registers with phone + OTP (lighter KYC than vendor)
  • Add to cart, place order, view order status
  • In-app messaging between buyer and vendor
  • Order tracking after dispatch

Payments & commissions

  • Paystack or Flutterwave integration — NGN only at launch
  • Escrow: funds held until buyer confirms delivery
  • Platform deducts 3–5% commission on release
  • Vendor payout: T+2 to registered bank account
  • Transaction receipts emailed and in-app
MVP 1 deliverables (as a developer)

Flutter mobile app (Android first, iOS alongside)

Buyer flow + vendor dashboard as separate tab/role views in one app

React admin web dashboard

Internal team only: vendor approvals, dispute resolution, payout management, transaction oversight

Node.js REST API (monolith to start, modular for later split)

Auth, users, listings, orders, payments, notifications — all in one deployable initially

PostgreSQL schema — users, vendors, listings, orders, transactions, disputes

Designed for scale from day one: UUIDs, soft deletes, audit timestamps

SMS OTP service (Termii or AfricasTalking)

Nigerian phone number verification at registration and on login

BVN/NIN verification integration

Mono, Okra, or Smile Identity API — budget ~₦50–80/verification

Push notifications (Firebase FCM)

Order updates, message alerts, payout confirmations

Basic in-app messaging system

Buyer ↔ vendor only. No group chats in MVP 1. Socket.io for real-time
What is NOT in MVP 1
  • No AI recommendations — add in Phase 2
  • No subscription tiers — launch free, monetize via commissions only
  • No ads or featured listings — Phase 2
  • No Bartar commodity exchange — that is MVP 2
  • No cross-border/export features — Phase 3
  • No logistics partner integration — vendor arranges own delivery for MVP, platform just tracks
  • No web storefront for vendors — mobile app only initially

M2

MVP 2 — Bartar commodity exchange (Q3 2027)

Built after Seed is stable and generating data. Bartar is a different product — it's enterprise B2B. The buyer is a trader or exporter, the seller is a cooperative or large producer, and the transaction size is 10–100x larger. Different UX, different KYC, different trust infrastructure.

Target: Q3 2027 launch → commodity exchange rollout
What MVP 2 must do

Commodity listings & RFQs

  • Sellers post bulk commodity offers: commodity type, grade, quantity, location, price/kg or per tonne
  • Buyers post Request for Quotes (RFQs) — system matches to sellers
  • Supported commodities at launch: maize, rice, sesame, cassava, cocoa, ginger, groundnuts
  • Offer expiry, minimum lot sizes, delivery terms

Enterprise KYC & verification

  • CAC registration number verification
  • NEPC export license verification for exporters
  • Cooperative registration documents
  • Director BVN/NIN cross-check
  • Manual review by Bia'net compliance team before approval

Trade coordination workflow

  • Offer → Negotiation → Deal confirmation → Contract generation
  • Digital trade contracts with PDF export
  • Milestone-based payment release (deposit, delivery, confirmation)
  • Dispute escalation with document evidence upload
  • Audit trail on every action — legally defensible

Market intelligence dashboard

  • Live commodity price feeds by region and grade
  • Historical price charts (weekly, monthly, seasonal)
  • Supply and demand indicators per commodity
  • Exportable reports for enterprise subscribers
  • Data sourced from platform transactions + external feeds
MVP 2 deliverables (as a developer)

Bartar web application (React) — enterprise buyers/sellers prefer desktop

Mobile companion app added post-launch. Web-first for Bartar unlike Seed

RFQ and offer matching engine

Algorithm matches RFQs to active offers based on commodity type, quantity, location, grade. PostgreSQL full-text search + scoring

Contract generation system

Template-based PDF contract auto-populated from deal terms. Node.js + PDFKit or Puppeteer

Escrow and milestone payment system

More complex than Seed — multiple release triggers, partial payouts, hold periods. Separate payment service module

CAC/NEPC verification integration

NIMC for NIN, CAC API for company verification, NEPC export license check (manual initially if no API available)

Market intelligence data pipeline

Python ETL jobs aggregating transaction data into price feeds. PostgreSQL time-series tables. Recharts on the frontend

Bartar notification system

Email (Sendgrid/AWS SES) + in-app. More email-heavy than Seed since users are enterprise, not mobile-first

Backend architecture & API design

Start as a modular monolith on Node.js (Fastify preferred over Express for performance). Each domain is a self-contained module. When transaction volume justifies it (around Phase 3), split high-load modules into microservices. This avoids premature complexity while keeping the future path open.

Monolith → modular → microservices on scale
Core API modules

Auth service

  • POST /auth/register → phone + OTP flow
  • POST /auth/verify-otp → issue JWT + refresh token
  • POST /auth/login
  • POST /auth/refresh
  • POST /auth/logout (revoke refresh token)
  • POST /auth/mfa/setup and /verify

User & vendor service

  • POST /vendors/apply — submit vendor application
  • PATCH /vendors/:id/approve — admin only
  • GET /vendors/:id/profile
  • POST /kyc/bvn-verify, /kyc/nin-verify
  • POST /kyc/face-check — third-party liveness API proxy
  • GET /users/:id/dashboard — aggregated stats

Listings service (Seed)

  • POST /listings — create new product listing
  • GET /listings — search with filters (category, location, price range)
  • GET /listings/:id
  • PATCH /listings/:id — update listing
  • DELETE /listings/:id — soft delete
  • POST /listings/:id/images — upload up to 5 photos (S3)

Orders service

  • POST /orders — create order from cart
  • GET /orders/:id — order detail + status
  • PATCH /orders/:id/status — vendor confirms, dispatches
  • POST /orders/:id/confirm-delivery — buyer confirms
  • POST /orders/:id/dispute — raise dispute
  • Webhooks: order.created, order.paid, order.delivered

Payments service

  • POST /payments/initiate — returns payment link (Paystack)
  • POST /payments/webhook — Paystack callback handler
  • POST /payments/escrow/release — trigger on delivery confirm
  • POST /payments/payout — vendor withdrawal to bank
  • GET /payments/transactions — ledger for user or vendor
  • Commission deduction logic runs here, not in orders service

Commodity service (Bartar)

  • POST /commodities/offers — create bulk offer
  • POST /commodities/rfq — post request for quote
  • GET /commodities/match/:rfqId — get matched offers
  • POST /deals — initiate deal from offer + RFQ match
  • PATCH /deals/:id/confirm — both parties confirm
  • GET /commodities/prices — live price feed by commodity + region
Key database schemas
-- Users table (shared across Seed and Bartar)
users: id (uuid), phone, email, role (buyer|vendor|enterprise|admin),
  kyc_status (pending|verified|rejected), bvn_verified (bool),
  nin_verified (bool), face_verified (bool), created_at, updated_at

-- Vendor profiles
vendors: id, user_id (fk), business_name, category, state, lga,
  bank_account_no, bank_code, approval_status, rating, total_sales

-- Product listings
listings: id, vendor_id (fk), title, category, price, unit, moq,
  stock_status, description, images (jsonb), location, deleted_at

-- Orders
orders: id, buyer_id, vendor_id, listing_id, quantity, unit_price,
  total_amount, commission_amount, status, delivery_address, dispute_id

-- Commodity deals (Bartar)
deals: id, offer_id, rfq_id, seller_id, buyer_id, commodity, grade,
  quantity_kg, price_per_kg, total_value, contract_url, milestones (jsonb),
  status (negotiating|confirmed|in_transit|completed|disputed)
Critical API endpoints — MVP 1 day-1 list
POST/auth/register

Phone number + OTP initiation. Required before any other action.

POST/vendors/apply

Vendor submits business details + KYC docs. Triggers BVN/NIN verification calls.

GET/listings?category=&location=&minPrice=&maxPrice=

Public endpoint. Full-text search + filter. Paginated. Redis cache on popular queries.

POST/orders

Authenticated buyer only. Creates order, initiates escrow payment hold via Paystack.

POST/orders/:id/confirm-delivery

Buyer confirms. Triggers automatic escrow release and vendor payout scheduling.

POST/payments/webhook

Paystack webhook. Must be idempotent — use payment reference as dedup key.

Infrastructure & DevOps

AWS as primary cloud. Start small, scale automatically. The key principle: design for scale from day one even if you deploy modestly at launch. Use managed services wherever possible — you don't have the team to run your own Kafka cluster or Redis Sentinel in Phase 1.

AWS-primary, Google Cloud for AI/ML services
AWS services to use

Compute

  • ECS Fargate — containerised Node.js API. No server management
  • EC2 (t3.medium) — Python AI/ML workers initially
  • Lambda — lightweight async jobs (commission calc, payout scheduling, email dispatch)
  • Auto Scaling Groups on ECS when traffic spikes

Data

  • RDS PostgreSQL (db.t3.medium, Multi-AZ for production)
  • MongoDB Atlas — unstructured data: chat messages, activity logs, product reviews
  • ElastiCache Redis — session tokens, listing search cache, rate limiting
  • S3 — product images, vendor documents, trade contracts (PDFs)
  • CloudFront CDN in front of S3 for image delivery

Messaging & queues

  • SQS — async job queues (payouts, KYC callbacks, notifications)
  • SNS — fan-out events to multiple consumers
  • EventBridge — internal domain events (order.created → payment service)
  • Socket.io on ECS for real-time chat and notifications

Observability

  • CloudWatch — logs, metrics, alarms
  • X-Ray — distributed tracing across services
  • Sentry — error tracking for Node.js and Flutter
  • Uptime checks — Route53 health checks on API endpoints
  • Dashboards for: transaction volume, error rates, payout status

Security

  • WAF (Web Application Firewall) — rate limiting, SQL injection protection
  • VPC with private subnets — DB never publicly accessible
  • Secrets Manager — API keys, DB credentials, payment keys
  • KMS encryption for data at rest (S3, RDS)
  • SSL/TLS everywhere — ACM certificates via ALB

CI/CD

  • GitHub Actions — automated test, lint, build on PR
  • Docker images pushed to ECR on merge to main
  • ECS rolling deployments — zero downtime updates
  • Staging environment mirrors production exactly
  • Database migrations: node-migrate or Flyway
Google Cloud — AI/ML layer
  • Vertex AI — hosted TensorFlow model training and serving for recommendation engine
  • Cloud Run — Python inference endpoints (recommendations API called by Node.js backend)
  • BigQuery — analytical queries on transaction data for market intelligence reports
  • Vision AI — supplementary image moderation for product listings (flags inappropriate uploads)

Technical risk register
High

KYC API reliability — BVN/NIN APIs (Mono, Okra, Smile Identity) have downtime

Mitigation: integrate two providers, fallback logic, manual review queue for failed verifications
High

Payment provider risks — Paystack/Flutterwave disputes, settlement delays

Mitigation: integrate both providers, maintain direct CBN settlement account, escrow reserve buffer
High

User adoption — vendors resistant to digital onboarding and KYC complexity

Mitigation: offline onboarding agents, simplified KYC flow, community ambassador program, local language support (Yoruba, Hausa, Igbo)
Medium

Regulatory changes — CBN fintech regulations, FCCPC marketplace rules

Mitigation: engage legal advisors proactively, CAC compliance, monitor CBN/SEC circulars, build flexible compliance module
Medium

Fraud and fake listings — fake vendors, product misrepresentation

Mitigation: manual admin review, face recognition, buyer dispute system, vendor rating system, automated ML fraud flag
Medium

Internet connectivity — target users in rural/peri-urban areas with poor data

Mitigation: Flutter offline-capable features for browsing, image compression, lightweight API responses, progressive loading
Low

Database performance at scale — PostgreSQL under high transaction volume

Mitigation: read replicas from day one, Redis caching on hot queries, connection pooling (PgBouncer), index strategy from schema design phase
Low

Team scaling — hiring backend/Flutter engineers in Nigeria with required seniority

Mitigation: remote-first hiring, competitive NGN salaries, equity participation, structured onboarding and documentation culture
